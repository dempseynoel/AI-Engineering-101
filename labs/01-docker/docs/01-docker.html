Docker for AI Engineers

Docker from scratch: build, containerise, and deploy a real sentiment analysis model end to end

Prerequisites

Docker Desktop installed and running.

Python 3.13+ for local development (the container itself bundles its own Python).

A code editor such as VS Code with a terminal you're comfortable using.

Git installed, to clone the seminar repo containing the demo source files.

Familiarity with Python and basic command line usage is assumed. No prior Docker or Azure experience is needed - that's what we're here to learn!

Lab Overview

This lab introduces Docker as the foundational tool for packaging and deploying AI/ML applications and models. As we data scientists transition into AI and ML engineering roles for our clients, the ability to ship models and whole applications as reliable and reproducible services becomes essential.

By the end of the lab we'll have used Docker to package a FastAPI sentiment analysis API (serving a relatively simple encoder only model) into a container, deployed it to Azure, and have it as a live endpoint.

We'll build the Dockerfile from scratch using multi-stage builds and uv for fast dependency installs, test locally, then push to Azure Container Registry and deploy on Azure Container Apps.

Learning Outcomes

By the end of this seminar, we'll be able to:

  • Explain what Docker is and why it matters for AI engineering
  • Understand how a real model is integrated into a serving application
  • Write a Dockerfile for an AI/ML application with model dependencies
  • Build, run, and deploy a containerised FastAPI model endpoint to Azure
Docker vs Containerisation

We're using Docker and containerisation interchangeably here, although they aren't the same thing. Containerisation is the concept of isolating an application and its dependencies into a portable, self-contained unit that runs consistently across environments. Docker is a specific tool that implements that concept and made it accessible to the mainstream although there are alternatives, e.g. Podman. Just remember that when we talk about Docker, we're referring to the whole concept of containerisation through Docker's specific approach.


1
What is Docker, and why should I care?

The core problem: "But it works on my machine..."

Imagine you bake a cake in your kitchen and it turns out perfectly. You hand someone the recipe, but their oven runs hotter, their flour is a different brand, and they're missing one ingredient. They follow the same steps and get a completely different result. The process of trying to reliably deploy code, whether that's an application or a full AI and ML model, into production from your local development environment can be much like this cake analogy.



A common problem we face arises when code behaves differently across machines due to mismatched Python or library versions. It might work on your machine, but not on someone else's. Things can get really complicated where projects have complex software stacks with strict version requirements such as TensorFlow tied to specific CUDA releases, or PyTorch conflicting with certain NumPy versions.

The solution? Ship the entire kitchen - oven, ingredients, and all - so the cake comes out the same every time, no matter where it's made. By containerising and packaging our code, libraries, and system dependencies Docker enables exact recreation of our runtime environments ensuring reproducibility wherever we run our code.

To summarise then, Docker helps with:

  • Reproducibility: same Python version, same library versions, same OS, everywhere
  • Dependency isolation: your PyTorch 2.x app won't clash with another team's TensorFlow 1.x service
  • Environment parity: dev, staging, and production all run the exact same container
  • Scalability: containers are the unit of scaling in Kubernetes, Azure Container Apps, ECS, etc
Why not just use venv or conda?

Because virtual environments only isolate Python packages. They do not isolate system libraries (like CUDA drivers), operating system differences (Windows vs Linux), or system level dependencies.

Core Docker Architecture

To containerise any of our code we need to understand some of the basics of how the Docker architecture interacts with each other: the Docker Engine (aka the daemon), the Docker CLI, and a registry.

The Docker CLI (client) sends commands to the Docker Engine (daemon), which does the heavy lifting of building images, starting containers, managing networks and volumes. When you type docker build or docker run, the CLI makes an API call to the daemon, which executes it and returns the result. Images are stored locally on your machine after being built or pulled, and can be pushed to a registry like Docker Hub or Azure Container Registry for sharing.

What's a daemon?

Pronounced "dee-mon". The background process that manages Docker objects - it listens for API requests from the CLI, and handles building images, running containers, managing networks, and storing volumes. You never interact with it directly; the CLI talks to it for you.

Docker Desktop bundles the Engine and CLI together and gives you a GUI on top, which is what most people install locally (what we've used here), but underneath it's still the same CLI-to-daemon relationship.

Most of the time though, you'll be focusing on these three core elements and concepts of Docker: dockerfiles, images, and containers.



What about a Virtual Machine (VM)?

Virtual machines and containers both solve the problem of running software in an isolated environment, but they do it very differently. A VM virtualises an entire operating system. It'll have a full guest OS, which means each VM carries its own kernel, system libraries, and binaries. This makes VMs heavy, typically gigabytes in size and taking minutes to boot. Containers take a lighter approach: they share the host machine's kernel and only package the application and its specific dependencies. The result is an isolated process that's megabytes rather than gigabytes, starts in seconds, and can be moved between machines with almost no friction.

2
Dockerfiles

What are they?

This is a text file that contains all the instructions and commands needed to build an image. This file adheres to specific format and set of instruction to build the required docker image. But a Dockerfile is not just a setup script - it's a contract.

The Dockerfile explicitly answers the question: "What must exist for my application to run?"

When you write a Dockerfile, you are documenting your assumptions in executable form. Every instruction in a Dockerfile exists because something once broke without it. Missing system libraries. Wrong Python version. A dependency that worked locally but failed in CI. Dockerfiles are written in the blood of past outages and failed builds.

Anatomy of a Dockerfile

When we write a Dockerfile, we've got to think about how Docker builds images in layers, and how each instruction in our Dockerfile creates a new one. When you rebuild an image, Docker checks each layer from top to bottom: if nothing has changed in that layer or any layer before it, it reuses the cached version and skips the work. The moment something changes, that layer and everything below it gets rebuilt from scratch. This is why ordering your Dockerfile carefully can mean the difference between a five second rebuild and a five minute one.

We'll be exploring our own Dockerfile in a moment, but in the meantime here are some of the key commands you'll almost always come across in any Dockerfile.

FROM

Every Dockerfile starts with this command which sets the base image you're building on top of. You'll typically be pulling a base image from somewhere like Docker Hub and its list of official images. Without a base image, you'd have to set up the operating system, install Python, configure paths, and handle all the low-level plumbing yourself.

Note

Base images range from bare-bones to full-featured. alpine is tiny (around 5MB) but can cause compatibility issues with Python packages that expect standard Linux libraries. slim variants like python:3.xx-slim strike a good balance. They're small enough to keep your image lean, complete enough that most packages install without fuss. The full python images include compilers and build tools, which is useful if your dependencies need to compile C extensions during install, but adds significant size.

WORKDIR

Sets the working directory inside the container. Every command that follows runs relative to this path. Think of it like running cd /path/to/somewhere on the command line.

COPY

Brings files from your local machine into the image, and this is where ordering the image layers becomes critical. Typically, we deliberately want to copy our package dependencies, e.g. requirements.txt, and install them on their own before copying the rest of our application code.

The reason: once we've copied in the requirements and installed them Docker will cache that entire installation layer. As long as our requirements.txt hasn't changed, Docker skips the install entirely on subsequent builds.

Think about your layering!

If we copied all the code first and then installed dependencies, changing a single line of application code (no matter how small or trivial) would invalidate the entire cache and force a full reinstall of every package which can take a very long time to rebuild.

RUN

Executes commands during the build process. This is where you install dependencies, e.g. RUN uv pip install -r requirements.txt, install system packages (RUN apt-get install), or run any setup scripts. Each RUN instruction creates a layer, so it's common to chain related commands together with && to keep the layer count down.

EXPOSE

Documents which port your application listens on. It doesn't actually publish the port but it signals to anyone reading the Dockerfile (or to orchestration tools) what port to expect.

Container Ports

Think of a container as a room with no doors. The app inside is listening on a given port that we expose (very commonly port 8000), but nothing can reach it. When we eventually run our container we need to cut a door between our machine and the container, connecting the two. We do this with the CLI command docker run -p our_port_number:the_container_port_number. It's common for these match, e.g. 8000:8000, but they don't have to. Just remember that the port on the right side has to match what you say is exposed in the Dockerfile.

CMD

Defines the default command that runs when a container starts from the image. Very commonly, this will be some sort of command that actually begins your application.

3
Dockerfiles: things to aim for

Try for as small an image as possible

When writing your Dockerfile and building the resulting image, you should aim for it to be as small and lightweight as you can get it. Here's a few reasons why:

Faster deployments

When you deploy a new container with an image Docker has to first pull the image before it can start the container. A 500MB image pulls in seconds. A 5GB image takes minutes. If we want to build scalable solutions, we need to think about how fast we can scale our applications out to meet user demand. The smaller the image, the quicker new containers can spin up.

Lower costs

Container images sit in a registry. You're paying for that storage. You're also paying for bandwidth every time an image gets pulled. If you have a CI pipeline that builds and pushes on every commit, and you're running across multiple environments (dev, staging, production), those costs add up. A smaller image is cheaper to store and transfer.

Smaller attack surface

Every piece of software in your image is a potential vulnerability. If your image includes a C compiler, and a CVE comes out for that compiler, your image is now flagged in security scans even though you never use the compiler at runtime. You have to rebuild and redeploy to clear the finding, or justify to your security team why it's there. If it's not in the image in the first place, the problem doesn't exist.

Multi-stage builds

If we're aiming for as small an image as possible, then multi-stage builds are one of the best ways to get there.

Think of it like packing for a holiday.

To decide what to bring, you need to spread everything out. You open the wardrobe, pull out all your options, lay them across the bed. You try things on, compare outfits, check the weather forecast. You might need scissors to cut a tag off something, an iron for a creased shirt. The bedroom is a mess, and that's fine.

But when it's time to actually pack the suitcase, you pick out the five or six things you're actually going to wear, fold them up, and put them in the bag. The scissors go back in the drawer. The iron goes back in the cupboard. The rejected outfits go back in the wardrobe. Your suitcase is small, light, and contains exactly what you need for the trip. Nobody looks at a packed suitcase and thinks "this should also contain the iron, just in case".



A single-stage Docker build is like zipping up the entire bedroom and checking it as luggage. Everything you touched during the process comes with you: the tools, the rejected options, the mess. It's heavy, it's expensive to transport, and most of what's in there you'll never use. But because of how Docker layers work, you can't really take things back out once they're in. Even if you remove them in a later step, they're still in the bag, just hidden under other stuff. You're still paying for the weight.

Multi-stage builds are the normal, sensible thing. Stage one is the bedroom where you spread everything out, use whatever tools you need, do the work. Stage two is the suitcase where you start empty, put in only what you need, and walk out the door. The mess in the bedroom doesn't follow you because the suitcase was never in the bedroom. It's a completely separate container that you packed deliberately.

Running as non-root

By default, when you run a container, everything inside it runs as root. Most people don't realise this because they never explicitly set it. Root is just the default, and it works, so nobody questions it.

The problem is that root inside a container is the same root as outside the container. Docker tries to isolate as much as possible but that isolation isn't perfect. Container escape vulnerabilities exist and they come up regularly in CVE lists. If someone exploits a bug in your application and they land inside the container as root, they have a much better starting position to break out of the container and into the host machine.

Creating a non-root user to run your application from is trivial. You create a user, make sure they own the files they need to access, and switch to them. That's it.

Scan for CVEs

Container scanning tools, for example Docker Scout analyse your image and flag known vulnerabilities in every package installed. OS-level packages, Python libraries, system utilities, all of it.

The thing people don't expect the first time they run a scan is how many vulnerabilities come back. Even a basic python:3.xx-slim image will have dozens of findings, most of them in OS packages you've never heard of and your application never touches. A full python:3.xx image will have hundreds. And if you're deploying a single stage build with all your build tools still in there, you'll get a report that's genuinely overwhelming (trust me!) and full of critical and high severity findings in packages that exist in the image for no reason.

This is where everything connects. Multi-stage builds will reduce the number of packages in your final image, which directly reduces the number of things a scanner can flag. Using as small an image as possible means fewer OS packages to begin with.


Tip

None of this is about passing a scan with zero findings. That's nearly impossible and not really the goal. It's about keeping the noise manageable and liaising with relevant security teams to triage flagged vulnerabilities.


Also be mindful that while having a smaller image (and by extension less "stuff" in it) can lead to fewer CVEs, this isn't always the case. If you remove large dependencies from the image it can cause the remaining packages to resolve differently in their dependency trees.


You could investigate using hardened base images which are deliberately stripped down to contain only what's absolutely necessary although the trade off is usability, meaning your Dockerfile has to be more precise because you can't rely on common utilities being there.

Docker Images

What are they?

A Docker image is built automatically after executing the instructions from a dockerfile. It's a frozen blueprint of your application. It contains everything your app needs to run - your code, runtime, libraries, system tools, and configuration defaults - but it doesn't actually do anything on its own. It's not alive.

Tip

Think of a Docker image like a restaurant recipe card. It lists ingredients, steps, and instructions. You can photocopy it a thousand times, but until you cook the dish, nothing happens. That cooking step is what creates a container.

What makes images powerful is that they are immutable. Once built, they never change. That immutability is the real reason Docker works so well in production. If the same image runs on my laptop, on a staging server, and on production, then "it worked on my machine" stops being an excuse.

Where do they live?

Images live in registries. When you run docker push in the CLI you upload your image to a registry like Docker Hub or more likely something like Azure Container Registry. When someone else or a pipeline runs docker pull, they/it download it. This is how you move images from your laptop to a server, or share them across a team. The image is identical everywhere - that's the whole point!

What are image tags?

A tag is a label you stick on an image to identify a specific version. This is done when you actually build the image. Tags are how we keep track of different versions. You might have my-image:v1, my-image:v2, and my-image:dev all sitting on your machine. Each points to a different immutable build. When you deploy, you specify which tag to use, for example my-image:v1 in staging, my-image:v2 in production so you always know exactly what's running where.

Latest Tag Gotchas!

An image with a tag of latest doesn't actually mean "the most recent build." It's just the default tag name. If you build my-image:v3 and then run docker pull my-image:latest, you'll get whatever was last tagged as latest, which might be weeks old. For anything beyond local development or demonstrations, use explicit version tags so there's no ambiguity.

Tip

Git based tags tie images directly to your source code. Consider using the commit hash as the tag for an image. The commit hash is the most reliable because it's unique and immutable. You can always trace an image back to the exact code that built it.

5
Docker Containers

What are they?

If an image is a blueprint, a container is a running instance of that blueprint. When you run a container, Docker takes the image, adds a thin writable layer on top, isolates it from the rest of your system, and starts executing the defined command. That's it.

You can run many containers from the same image, and they don't interfere with each other. This is how modern scaling works. Instead of making one giant server stronger, you run ten identical containers behind a load balancer. If one dies, you replace it. No emotional attachment.

How our API works

API quick walkthrough

This is a very quick summary of how we're using FastAPI with a single endpoint: POST /predict for our demo. You send it text, you get back a label (positive/negative) and a confidence score. That's it.

The DistilBERT model wrapper lives in its own module. It loads a tokenizer and an ONNX Runtime session at startup. If anything goes wrong during loading, the app crashes immediately rather than running in a broken state.

The inference code tokenizes the input, runs it through the ONNX session, and returns the top label.

Note

ONNX stands for Open Neural Network Exchange. It's an open format for representing machine learning models. Essentially a common file format that different ML frameworks can all read and write. The problem it solves is that models are typically trained in a specific framework like PyTorch or TensorFlow, and those frameworks are large and complex. PyTorch alone is over 2GB. ONNX lets you export a trained model into a framework-agnostic format.

Why are we using DistilBERT?

In a nutshell, because it's a real model (one of the most popular on HuggingFace) that sits in the sweet spot for teaching deployment. It's also not a toy scikit-learn classifier so the Docker and infrastructure patterns should be transferable to real work. It's also small enough to run on CPU, so we don't need to worry about GPUs, model sharding, or any of the complexity that comes with serving full LLMs ourselves.

Note

In practice, most AI engineering work today involves calling a model through an API endpoint, whether that's calling GPT, Claude, or Gemini through a managed API or similar managed services, rather than downloading and serving a model yourself. You send text in, you get a response back, and the provider handles the infrastructure for the model. Images based on this pattern will usually be very small as there's no baked in model weights.


We've deliberately taken a different approach here by downloading a pre-trained model and packaging it into our container because it surfaces more of the engineering decisions you could face in production: dependency management, image size, multi-stage builds, and runtime optimisation.


These concerns don't disappear when you're calling an API, but they're easier to overlook. By working closer to the model, we get to confront them directly, and the patterns transfer to any containerised application, whether it's serving a model or wrapping an API.

Writing our APIs Dockerfile

Attempt 1: Simple version

I wrote the simplest possible Dockerfile first. Install PyTorch, load the model, serve it. It worked. But the image was 9.4GB.

Why was it big? Well, the default PyTorch install from PyPI pulls in CUDA libraries - GPU drivers, cuDNN, the full NVIDIA runtime stack - even if you're deploying to a CPU only container. I was putting in hundreds of megabytes of GPU tooling that will literally never execute. Having an image this would mean slow builds (nearly 10 minutes), slow pulls, expensive storage and sluggish scale outs.

Note

The DistilBERT model we are using is a PyTorch model (it was trained in PyTorch, its weights are stored in PyTorch's format, and loading it with Hugging Face's transformers library spins up a PyTorch model object) I knew I definitely needed PyTorch at runtime as this was the thing actually running the prediction.

Attempt 2: PyTorch CPU only

The first optimisation was to install torch with the CPU-only index URL so I didn't pull in CUDA. It was all the same code, the only change was how PyTorch got installed. It was much faster to build (2 minutes) and the image dropoed to 2.5GB!

That's a significant win for essentially zero code changes. But 2.5GB is still heavy, and the reason is that PyTorch itself is a large framework even without CUDA. I was still packing in an entire deep learning training framework just to run inference on a single small model. It's like bringing the whole kitchen to serve one dish.


Attempt 3: ONNX runtime

This where there was some actual effort, and to be honest some failed attempts locally. I couldn't just swap a dependency, I had to change the model loading code, manually write the inference code (i.e. not just using the transformers wrapper), update my Python version in the base image as well as updating dependencies and split my requirements.txt into build-time and runtime files.

The model gets exported to ONNX format during the Docker build (in Stage 1, with PyTorch present), and the runtime image (Stage 2) only needs onnxruntime, which is a fraction of PyTorch's size. The final image droped to 1.5GB (a massive 85% reduction in size), and took only 90 seconds to build!



The point here is that this is a normal engineering process. You don't arrive at the optimised version on day one. You start with "make it work," then you look at what you've built and ask what's unnecessarily heavy. Each iteration involved a tradeoff.

Despite the reduction in size though, there's still vulnerabilities in the image. And really the only improvement has been the removal of the criticals and a few highs. In reality, this is where you'd go and investigate what they are, speak to the security folks to understand risk appetites and make decisions on how much you want to focus on reducing them further. As we've seen there's a pay off between trying to reduce image sizes, CVEs and potentially increasing complexity of how you write your Dockerfile and design the image to accomodate that.


Anatomy of a Dockerfile

The final Dockerfile show's the payoff of the multi-stage build discussed earlier. Stage 1 is the bedroom: it pulls in PyTorch, the Hugging Face export tooling, and everything else needed to convert the model to ONNX. None of that ships in the final built image from Stage 2.

FROM python:3.13-slim AS builder

A slim version of Python was chosen rather than alpine or the full image as the base, and the AS builder alias is what makes multi-stage work; it gives this stage a name so Stage 2 can selectively copy from it.

COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv

We pull the uv binary directly from a published container image rather than installing it with pip or curl and then having to cleanup.

COPY requirements-build.txt

RUN uv pip install

We copy the build requirements file first and install before copying any application code. If our application code changes but our dependencies don't, Docker skips the entire install on rebuild. This build time requirements has everything we need to convert and export the HuggingFace model into an ONNX model.

RUN python -c

We download the pretrained DistilBERT model from Hugging Face and convert it to ONNX format during the build, not at runtime. That means the final image ships a pre-exported ONNX model and never needs PyTorch, the optimum library, or network access to Hugging Face. It also means the model is baked into the image.

Baking models in the image

The main downside is that the model and the application code are coupled together in the same image. If I wanted to update the model (say fine tune it or swap to a newer version) I'd need have to rebuild and redeploy the entire image even if the application code hasn't changed at all.


The alternative would be to decouple them: store the model externally (cloud storage, a model registry, a mounted volume) and have the application pull it at startup. That would let us update the model independently of the code, and our image stays smaller. The tradeoff is we'd now have a startup dependency on an external service, our cold start time increases, and we'd need to handle what happens if that service is unreachable.

COPY requirements-runtime.txt

RUN uv pip install --prefix=/runtime

We copy the runtime requirements file which has all the packages we need at runtime, and we install them at the --prefix=/runtime so they are in an isolated directory rather than the system Python. This is so Stage 2 can copy just that directory and get a clean set of runtime only packages without any build contamination.

RUN useradd --create-home appuser

We create non-root user. A simple way to improve container security.

COPY --from=builder

This is where the multi-stage payoff becomes concrete. We selectively copy three things from the builder: the runtime Python packages (no PyTorch), the uvicorn binary, and the exported ONNX model. Everything else stays behind in Stage 1 and never enters the final image.

RUN chown

USER appuser

The chown ensures the non-root user actually has permission to read the model files that were copied in as root. Then USER switches all subsequent commands (including CMD) to run as appuser. Without the chown, the app would crash at startup with a permission denied error on the model directory.

How to build and run the image locally

Build

To get our model API built and running locally, we first need to make sure Docker Desktop is running and open a terminal in the directory containing our Dockerfile and application code. Everything from here happens through the Docker CLI. We'll type commands, the CLI passes them to the Docker Engine running in the background, and the engine does the work.

To build the image, run docker build -t bert-sentiment-api:version1 . This tells Docker to read the Dockerfile in the current directory, execute every instruction from top to bottom, and tag the resulting image as bert-sentiment-api:version1. The first build will take a few minutes because it's downloading the base image, installing dependencies, and exporting the model. Subsequent builds will be much faster thanks to layer caching, assuming you haven't changed your requirements files

The -t flag

Pay attention to the -t flag in the command we ran. It's this flag that tags the image with a name and version. bert-sentiment-api is the name, version1 is the tag. These are seperate by a : (colon). Without -t, Docker will still build the image but it'll only have a random hash as an identifier, which isn't particularly useful when we need to reference it later.


Also, notice the . at the end of the command. This tells Docker which directory to use as the source for any COPYinstructions in the Dockerfile. The . means "the current directory." Docker sends the entire contents of this directory to the engine, which is why a .dockerignore file matters.

We've not discussed this yet, but it's out in project directory and without one, we'd be sending everything, including things like .git folders, __pycache__, local virtual environments, and anything else sitting in the directory that the image doesn't need.

Run

Once the build finishes, run docker run --name bert-api -p 8000:8000 bert-sentiment-api:version1 to start a container from the image. The -p 8000:8000 flag maps port 8000 on your machine to port 8000 inside the container. Without this, the API would be running but unreachable. The --name flag just allows us to call our container a name, which will make it easier to stop in a moment (otherwise you'll get a default generated one). Once you see the Uvicorn startup message in your terminal, open http://localhost:8000/docs in a browser to get FastAPI's interactive Swagger UI, where you can test the /predict endpoint directly.


Stop

Once you've tried the local container, you can stop it with this command docker stop bert-api. If you don't stop the container, it'll just container running in the background and waste resources.

How to deploy to Azure

Azure Container Registry

Before we can deploy to Azure, the image needs to live somewhere Azure can pull it from. Azure Container Registry is a private registry hosted on Azure cloud that integrates with all Azure services - so this is a obvious choice if we are deploying to Azure, although there's nothing preventing us having the image stored elsewhere!

The following commands can be used to create the Azure Container Registry:

az login

First login to your Azure subscription, or Capgemini's if you plan on using the environment we've setup for the training series.

az acr create --name "yourregistryname" --resource-group "yourresourcegroup" --sku Basic

This creates a registry for us, where the yourregistryname and yourresourcegroup are the unique names you'll give for the registry and resource group.

az acr login --name yourregistryname

Once the registry exists, log Docker into it. From here we'll able to tag the image we built locally (and which is under 'images' on the Docker Desktop Dashboard) to point to our container registry.

docker tag bert-sentiment-api:version1 youregistryname.azurecr.io/bert-sentiment-api:version1

This tells the local image we built to point towards our Azure Container Registry.

docker push yourregistryname.azurecr.io/bert-sentiment-api:version1

Finally, we push the local image up to the Azure Container Registry.

Azure Container Apps

With the image in ACR, we can deploy it to Azure Container Apps, which is a managed service that runs our container without us having to provision or manage any underlying infrastructure. It's essentially a wrapper around Kubernetes which means we don't have to deal with the complexities of that, but which still allows for auto-scaling to meet demand, scaling to zero, and host of other capabilities.

The following commands can be used to create the Azure Container Apps service:

az containerapp env create --name yourenvname --resource-group yourresourcegroup --location uksouth

This creates our Container Apps environment which will can host multiple container apps.

az containerapp create --name bert-sentiment-api --resource-group yourresourcegroup --environment yourenvname --image yourregistryname.azurecr.io/bert-sentiment-api:version1 --target-port 8000 --ingress external --registry-server yourregistryname.azurecr.io

With the Container App environemnt provisioned, we can deploy an actual container app to sit within the environment. The --target-port 8000 tells Azure which port your app listens on (matching the EXPOSE in your Dockerfile), and --ingress external makes it publicly accessible.

Publicly accessible ports

This is almost certainly not what you would want to do with a client. Leaving URLs and ports open to the outside world is never a good idea. I'm only doing it here as setting up firewalls and netwroking permissions is complex and out of scope for this demonstration.

Once the deployment finishes, Azure gives you a URL. Open it with /docs on the end and you've got the same Swagger UI you had locally, now running as a live endpoint. The image running in Azure is identical to what you tested on your machine - same code, same model, same dependencies. That's the whole point

Delete your resources

Use the following command to delete your Azure reources, otherwise you may incur significant costs.

$ az group delete --name yourresourcegroup --yes --no-wait

What's next

Terraform and IAC

In this lab we deployed our containerised API by running Azure CLI commands manually in the terminal. That works for learning and one off deployments, but it doesn't scale to real projects. If you needed to recreate the same infrastructure in a second environment, or hand it off to a different team, they'd have to re-run the same sequence of commands and hope they got every flag right. There's no record of what exists, no way to review changes before they happen, no way to detect infrastructure drift, and no easy way to tear it down and rebuild it consistently.

In Lab 2 we'll solve that with Terraform, an infrastructure as code tool that lets you define your cloud resources: the container registry, the container apps environment, the container app itself all as declarative configuration files. Instead of running imperative commands one at a time, you describe what you want to exist and Terraform figures out how to make it happen. The same sentiment analysis API, the same Docker image, but deployed through code that can be version controlled, reviewed in a pull request, and reused across environments.
